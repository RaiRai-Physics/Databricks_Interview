11. What is Unity Catalog?

Answer:
Unity Catalog is Databricks’ governance layer for managing data and AI assets. It provides centralized access control, permissions, auditing, lineage, and governance across workspaces.

The hierarchy is usually:

Catalog → Schema → Table/View/Function/Volume

Example:

SELECT * FROM healthcare_catalog.project_core.claims;

Unity Catalog controls access using privileges, ownership, policies, and data-level controls.

12. What is the difference between a managed table and an external table?

Answer:

Managed Table	External Table
Databricks manages data and metadata	Databricks manages metadata only
Dropping table can remove data	Dropping table usually does not delete source data
Stored in managed storage location	Stored in external path like S3
Simpler for internal lakehouse use	Better when data is shared with external systems
13. What is Auto Loader?

Answer:
Auto Loader is a Databricks feature used to incrementally ingest new files from cloud storage. It is commonly used with S3, ADLS, or GCS.

It uses the cloudFiles source and can process newly arrived files efficiently without rereading everything.

Example:

df = spark.readStream.format("cloudFiles") \
  .option("cloudFiles.format", "csv") \
  .load("s3://bucket/input-data/")
14. What is Structured Streaming?

Answer:
Structured Streaming is Spark’s streaming engine. It allows us to process data continuously or incrementally using DataFrame APIs.

In Databricks, it is used for:

Real-time ingestion
Incremental ETL
Streaming reads and writes from Delta tables
Auto Loader pipelines

Databricks uses Structured Streaming for near real-time and incremental processing workloads.

15. What is checkpointing in streaming?

Answer:
Checkpointing stores streaming progress, metadata, and offsets. It helps Spark know what data has already been processed.

Without checkpointing, a streaming job may reprocess files or lose progress after failure.

Example:

df.writeStream \
  .format("delta") \
  .option("checkpointLocation", "s3://bucket/checkpoints/claims") \
  .toTable("claims_bronze")
16. What is Databricks Workflows or Lakeflow Jobs?

Answer:
Lakeflow Jobs, formerly commonly referred to as Databricks Jobs or Workflows, is used to schedule and orchestrate production workloads. It can run notebooks, Python scripts, SQL tasks, JARs, dbt tasks, and multiple dependent tasks in a workflow.

Databricks describes Lakeflow Jobs as workflow automation for coordinating and scheduling data processing workloads.

17. What is Delta Live Tables?

Answer:
Delta Live Tables, or DLT, has been updated under the newer name Lakeflow Spark Declarative Pipelines. It is used to build reliable batch and streaming pipelines using SQL or Python.

Important interview note:
“DLT code still works, but Databricks now refers to this capability as Lakeflow Spark Declarative Pipelines.”

Example concept:

import dlt

@dlt.table
def clean_claims():
    return spark.read.table("raw_claims").dropDuplicates()

18. What is the difference between batch and streaming?

Answer:

Batch	Streaming
Processes fixed data	Processes continuous/incremental data
Runs on schedule	Runs continuously or trigger-based
Example: daily claims load	Example: new claim files arriving every few minutes
Easier to debug	Better for low-latency pipelines

19. What is a cluster in Databricks?

Answer:
A cluster is a group of compute resources used to run Spark workloads. In Databricks, compute can be all-purpose, job compute, serverless compute, or SQL warehouses depending on workload type.
Common types:

All-purpose cluster: interactive notebooks
Job cluster: automated jobs
SQL warehouse: SQL queries and dashboards
Serverless compute: Databricks-managed compute

20. What is the difference between all-purpose cluster and job cluster?
Answer:

All-purpose Cluster	Job Cluster
Used interactively	Used for scheduled jobs
Shared by users	Created for a specific job
Can stay running	Usually terminates after job
Good for development	Better for production and cost control
