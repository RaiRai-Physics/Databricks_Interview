21. How do you improve performance in Databricks?

Answer:
I would check:

File size, avoid too many small files
Partition strategy
Use Delta format
Use OPTIMIZE for compaction
Use liquid clustering or Z-ordering depending on table design
Cache only when reused
Avoid wide shuffles
Broadcast small lookup tables
Tune cluster size
Filter early and select only needed columns

Databricks recommends liquid clustering for new Delta tables, and OPTIMIZE rewrites data files to improve layout and query performance.

22. What is OPTIMIZE?

Answer:
OPTIMIZE is a Delta command that compacts many small files into larger files and improves data layout.

Example:

OPTIMIZE claims;

For large Delta tables, this can improve query performance because Spark scans fewer files.

23. What is Z-ordering?

Answer:
Z-ordering colocates related data in the same files based on selected columns, helping queries skip unnecessary data.

Example:

OPTIMIZE claims
ZORDER BY (patient_id, claim_date);

But for newer table designs, Databricks recommends liquid clustering instead of Z-ordering for new tables.

24. What is liquid clustering?

Answer:
Liquid clustering is a newer Databricks data layout optimization feature. It organizes table data using clustering keys and can replace traditional partitioning and Z-ordering for many use cases.

Interview line:
“Liquid clustering is useful when query patterns change over time because it gives more flexible table layout optimization than static partitioning.”

25. What is partitioning?

Answer:
Partitioning stores data in folder-like groups based on column values.

Example:

df.write.partitionBy("claim_year").format("delta").saveAsTable("claims")

Good partition columns have:

Low to moderate cardinality
Frequent filtering
Business meaning, such as date, region, or year

Bad partition column example: claim_id, because it has too many unique values.

26. What is a shuffle?

Answer:
A shuffle happens when Spark redistributes data across partitions, usually during joins, aggregations, groupBy, distinct, or orderBy.

Shuffles are expensive because they move data across the network.

Ways to reduce shuffles:

df.select("claim_id", "amount").filter("amount > 1000")

Filter early, avoid unnecessary columns, and use broadcast joins for small tables.

27. What is a broadcast join?

Answer:
A broadcast join sends a small DataFrame to all worker nodes so Spark does not need to shuffle a large table.

Example:

from pyspark.sql.functions import broadcast

result = claims.join(broadcast(disease), "disease_id")

Use it when one table is small enough to fit in memory.

28. What is the difference between repartition() and coalesce()?

Answer:

repartition()	coalesce()
Increases or decreases partitions	Usually decreases partitions
Causes full shuffle	Avoids full shuffle when reducing
More balanced	Faster but less balanced
Use before heavy parallel writes	Use before reducing output files

Example:

df.repartition(10)
df.coalesce(1)
29. How do you handle null values in PySpark?

Answer:

df.select([count(when(col(c).isNull(), c)).alias(c) for c in df.columns])

Replace nulls:

df.fillna({"gender": "NA", "city": "Unknown"})

Drop nulls:

df.dropna(subset=["patient_id"])

In an interview, say:
“I first profile nulls by column, then decide whether to drop, replace, or flag them based on business rules.”

30. How do you remove duplicates in PySpark?

Answer:

df.dropDuplicates()

For business-key deduplication:

df.dropDuplicates(["claim_id"])

For latest record per key:

from pyspark.sql.window import Window
from pyspark.sql.functions import row_number, desc

w = Window.partitionBy("claim_id").orderBy(desc("updated_at"))

df_latest = df.withColumn("rn", row_number().over(w)) \
              .filter("rn = 1") \
              .drop("rn")
